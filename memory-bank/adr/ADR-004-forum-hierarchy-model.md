# ADR-004: Hierarchical Forum Data Model (sections → topics → posts)

## Контекст

При реализации форума (FT-005) нужно выбрать модель организации контента. Форум LFC.ru предполагает разделение по тематическим разделам (матчи, трансферы, жизнь клуба и т.д.) — выбор модели данных определяет схему БД, маршрутизацию и ролевой контроль.

## Драйверы решения

- Стандартный пользовательский ожидаемый опыт для форума: разделы → темы → посты
- Ролевое разграничение: гость читает, пользователь пишет посты, модератор управляет структурой (разделами и темами)
- Маршруты `/forum`, `/forum/sections/:id`, `/forum/topics/:id` заданы в требованиях
- Простота реализации на PostgreSQL + Go без сложных рекурсивных структур

## Рассмотренные варианты

| Вариант | Плюсы | Минусы | Статус |
|---|---|---|---|
| Иерархическая (sections→topics→posts) | Привычна пользователям форумов, чёткое ролевое разграничение, простые JOIN-запросы | Фиксированная глубина 3 уровня, нет кросс-раздельных тем | **Выбрано** |
| Плоская (один уровень тем) | Проще схема | Нет организации по разделам, теряется навигация | Отклонено |
| Тегированная (topics + теги) | Гибкость, кросс-тематические темы | Сложнее навигация, нестандартный UX для фан-форума | Отклонено |

## Решение

Форум использует трёхуровневую иерархию: `sections` → `topics` → `posts`. Каждая сущность — отдельная таблица в PostgreSQL. Разделы и темы создаются ролями `Moderator`/`Admin`; посты — ролью `User` и выше.

Схема:
- `sections(id, name, description, created_at)`
- `topics(id, section_id FK, title, author_id FK, created_at)`
- `posts(id, topic_id FK, author_id FK, content, created_at, quote_post_id FK nullable)`

## Последствия

### Положительные

- Стандартная форумная структура — пользователи понимают навигацию без объяснений
- Чёткое разграничение прав: разделы/темы — mod/admin, посты — user
- Простые JOIN-запросы для счётчиков и метаданных

### Отрицательные

- Фиксированная глубина: нельзя создать подразделы без изменения схемы
- Тема может существовать только в одном разделе (нет кросс-постинга)

### Нейтральные / организационные

- Необходимы миграции для таблиц `sections`, `topics`, `posts`
- Маршруты `/forum`, `/forum/sections/:id`, `/forum/topics/:id` фиксируются как API-контракт

## Риски и mitigation

| Риск | Mitigation |
|---|---|
| Потребность в подразделах в будущем | При возникновении потребности — отдельный ADR; текущая схема не закрывает такую эволюцию |
| Большой объём постов в одной теме | Пагинация на стороне `/forum/topics/:id`; пост-MVP |
