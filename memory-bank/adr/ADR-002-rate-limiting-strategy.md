# ADR-002: IP-based Rate Limiting on Login

## Контекст

Эндпоинт `/login` должен быть защищён от автоматизированного подбора паролей (REQ-4 из FT-001). Нужно выбрать стратегию ограничения: по IP-адресу, по идентификатору пользователя или по токену запроса.

`PCON-03` в `domain/problem.md` фиксирует обязательность rate limiting, но не определяет стратегию.

## Драйверы решения

- Rate limiting должен срабатывать до успешной аутентификации (атакующий не авторизован)
- Реализация через `ratelimit`-пакет (уже принятый инструментарий проекта)
- Порог: 5 попыток / 60 сек / IP → блокировка 10 мин (бизнес-требование из FT-001 REQ-4)
- Минимальная сложность реализации на MVP

## Рассмотренные варианты

| Вариант | Плюсы | Минусы | Статус |
|---|---|---|---|
| IP-based | Применяется до аутентификации, простая реализация | Блокирует пользователей за NAT/shared proxy | **Выбрано** |
| User-based (по email) | Точнее, не затрагивает других пользователей | Требует lookup пользователя до блокировки; обходится через разные аккаунты | Отклонено |
| Token-based (CAPTCHA) | Высокая точность | Сложность UX, внешняя зависимость | Отклонено для MVP |

## Решение

Rate limiting реализован на уровне IP-адреса через middleware перед обработкой `/login`. Порог: >5 неудачных попыток за 60 сек → HTTP 429, блокировка IP на 10 минут. Реализуется через `ratelimit`-пакет.

## Последствия

### Положительные

- Срабатывает до аутентификации — нет необходимости в lookup пользователя
- Простая реализация через middleware
- Совместим с `PCON-03` из `domain/problem.md`

### Отрицательные

- Пользователи за общим NAT или corporate proxy могут быть заблокированы из-за действий других
- Злоумышленник за уникальным IP не ограничен блокировкой других IP

### Нейтральные / организационные

- Порог (5 попыток / 60 сек) и время блокировки (10 мин) — конфигурируемые параметры

## Риски и mitigation

| Риск | Mitigation |
|---|---|
| Ложная блокировка пользователей за общим NAT | Приемлемо для MVP; при необходимости — добавить user-based слой поверх |
| Обход через смену IP | Допустимо для MVP-уровня защиты; полное решение требует CAPTCHA |

## Follow-up

- Реализовать `ratelimit`-middleware на `/login`
- Покрыть интеграционным тестом: SC-4 из FT-001
