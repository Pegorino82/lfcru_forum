# ADR-005: Хранение изображений статей — файловая система + Docker volume

## Контекст

Для статей в admin-панели нужна загрузка изображений. Требуется решить, где физически хранить загружаемые файлы: в БД, на облачном хранилище или на файловой системе.

## Драйверы решения

- Проект работает в Docker на собственном сервере; внешних облачных зависимостей нет и не планируется.
- Go-бэкенд доступен на хосте через примонтированные volumes — простая раздача файлов через nginx уже есть в инфраструктуре.
- Размер проекта (фан-сайт) не требует CDN или object storage на старте.
- Простота операционного контроля: файлы легко бэкапить, инспектировать, удалять вручную.

## Рассмотренные варианты

| Вариант | Плюсы | Минусы | Почему рассмотрен |
| --- | --- | --- | --- |
| Файловая система + Docker volume | Простота, нет зависимостей, nginx раздаёт статику | Не масштабируется горизонтально (нет shared FS) | **Выбран** — достаточен для текущего масштаба |
| PostgreSQL (bytea / large objects) | Единое место хранения, транзакционность | Раздувает БД, медленно для крупных файлов | Отклонён |
| S3-совместимое хранилище (MinIO/Yandex Object Storage) | Масштабируется, CDN-ready | Дополнительная инфраструктура, зависимость от внешнего сервиса | Отклонён как преждевременный |

## Решение

Изображения хранятся в директории на файловой системе хоста. Путь к директории задаётся через env-переменную `UPLOADS_DIR` (default: `./uploads`). Директория примонтирована в контейнер через Docker volume. nginx раздаёт файлы из этой директории как статику (маршрут `/uploads/*`).

При загрузке:
1. Файл нормализуется на стороне Go (изменение размера до max ширины, конвертация в WebP).
2. Сохраняется по пути `$UPLOADS_DIR/{article_id}/{uuid}.webp`.
3. В БД хранится относительный путь (или только имя файла) для формирования URL.

## Последствия

### Положительные

- Нет внешних зависимостей; работает в полностью offline-среде.
- nginx раздаёт файлы напрямую — Go не участвует в serving статики.
- Бэкап через rsync/docker cp.

### Отрицательные

- При горизонтальном масштабировании (несколько инстанций Go) нужен shared volume или миграция на object storage.
- Файлы и записи в БД могут рассинхронизироваться при ручном удалении файлов.

### Нейтральные / организационные

- `internal/config/config.go` — добавить `UploadsDir string`.
- `docker-compose.dev.yml` и `docker-compose.prod.yml` — добавить volume mount.
- `memory-bank/ops/config.md` — обновить после добавления `UPLOADS_DIR`.

## Риски и mitigation

- **Orphaned files** — файлы без записей в БД. Mitigation: при удалении статьи или изображения через admin-панель удалять файлы синхронно.
- **Disk full** — нет квот на загрузку. Mitigation: нормализация размера изображений снижает риск; в будущем можно добавить проверку свободного места.
